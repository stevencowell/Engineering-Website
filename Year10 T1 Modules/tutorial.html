Plan the System

Keep the engineering problem simple: what condition starts watering, what controls the decision, and what output delivers the water?

Input, process and output

  • Input: soil moisture sensor, rain sensor, push button, timer, switch or teacher-approved trigger.
  • Process: controller, relay, code, logic gate or manual control decision.
  • Output: pump, valve, LED warning, buzzer or watering mechanism.
  • Feedback: evidence that the system watered correctly, stopped correctly and can be tested again.

Sketch the Layout

Water path

  • Mark the reservoir, pump or valve, tubing route, outlet and plant/garden area.
  • Show possible leak points, spill zones and where the water should safely drain or collect.
  • Keep tubing short, supported and clear of moving parts or loose wires.

Electronics path

  • Mark the power source, controller, input device, output device and any relay or driver module.
  • Separate the wet zone from the dry electronics zone. This is not optional. Water and electrics are a crap friendship.
  • Label the parts so another student could trace the system without guessing.

Safe Test Setup

Before power is connected

  • Teacher checks the circuit, power source and water container position.
  • Wet zone and dry electronics zone are clearly separated.
  • Spill control is ready: tray, towel, container or sink area.
  • Tubing and wires are secured so they cannot pull loose during testing.
The best test setup is boring: controlled, repeatable and easy to clean up. Drama belongs in English, not near the power supply.

Write the Control Logic

Decision rule

  • Write the rule before building: if the input condition is met, the output turns on.
  • Include a stop condition: stop after a set time, when soil is wet, when the switch is off or when the teacher stops the test.
  • Use a flowchart or pseudocode so your thinking is visible in the folio.

Simple example

  • If soil is dry and rain is not detected, turn pump on for 5 seconds.
  • If soil is wet or rain is detected, keep pump off.
  • After watering, wait, recheck and record the result.

Build and Test Sequence

1. Output test Can the pump, valve or indicator work by itself?
2. Input test Does the sensor, switch or timer trigger reliably?
3. Full test Does the input control the output over repeated cycles?

Test data to record

  • Input condition: dry/wet reading, switch position, timer setting or manual trigger.
  • Response: output on/off, response time, water delivered and whether the system stopped.
  • Faults: leaks, loose wires, sensor errors, blocked tubing, weak battery or unreliable threshold.
  • Fixes: what changed, why it changed, and whether the retest improved the result.

Evidence to Capture

  • Photos: labelled layout, safe setup, build stages, final prototype and testing action shot.
  • Diagrams: input-process-output chain, circuit/control diagram, water path sketch and final layout.
  • Algorithm: flowchart or pseudocode showing the watering decision and stop condition.
  • Data: subsystem tests, full-system tests, response time, water amount, repeated-cycle reliability and fault log.
  • Evaluation: what worked, what failed, what changed, and one specific improvement supported by test evidence.

Evaluation Sentence Starters

  • Our system met the brief because the input triggered the output when...
  • The most reliable part of the system was... because the test data showed...
  • The main fault was... and we think this happened because...
  • After changing... the retest showed...
  • Next time I would improve... because this would make the system safer, more reliable or more water efficient.

Quick Q&A

What should I test first?
Test the output first. If the pump, valve or indicator cannot work by itself, the full system will not magically behave.
Why separate wet and dry zones?
It protects the circuit, reduces shock and short-circuit risk, and makes faults easier to trace during testing.
What counts as useful evidence?
Evidence shows the design decision, test result or improvement clearly: photos, labelled diagrams, algorithm, data table, fault log and retest notes.
What if the prototype fails?
Record the failure, identify the likely cause, fix one thing, and retest. A documented fault and fix is strong engineering evidence.
